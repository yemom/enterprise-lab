# Reflection

### 1. Why should the ProductRequest DTO carry the `@Valid` annotations instead of the Product entity itself?
Validating the DTO (Data Transfer Object) rather than the Entity enforces a strict separation of concerns and improves security. DTOs represent the precise data structure expected at the API boundary, while Entities map to the database schema. Validating at the DTO level ensures that invalid data is rejected at the entry point, before it ever reaches your domain logic. Furthermore, validation rules for incoming requests often differ from the constraints of a persisted entity (e.g., an ID might be required for an existing entity but prohibited when creating a new one). This approach also protects against mass assignment vulnerabilities by strictly controlling which fields a client is permitted to send.

### 2. What is the purpose of the Location header returned on a POST 201 Created response, and which HTTP specification mandates it?
The `Location` header is used to provide the client with the exact URI (URL) where the newly created resource can be found and accessed. This saves the client from having to construct the URL manually or perform a secondary search query to retrieve the new item. This behavior is mandated by the HTTP Semantics specification, currently defined in **RFC 9110** (which obsoleted the older RFC 7231 and RFC 2616 standards).

### 3. Explain the difference between `@ControllerAdvice` and `@ExceptionHandler`. When would you use each?
- **`@ExceptionHandler`**: This annotation is applied to methods within a specific `@Controller` or `@RestController`. It intercepts and handles exceptions thrown only by the methods within that specific controller class. You use this when you need a highly specific error response that only applies to one particular controller.
- **`@ControllerAdvice`**: This is a class-level annotation that provides global exception handling. Methods annotated with `@ExceptionHandler` inside a `@ControllerAdvice` class will intercept exceptions thrown by *any* controller across the entire application. You use this to establish a consistent, application-wide error handling strategy (for example, ensuring that all `ResourceNotFoundException`s return the same structured JSON format).

### 4. In your MockMvc tests you used `@Transactional` on the test class. What would happen to the database state between tests if you removed this annotation?
If `@Transactional` were removed, the database changes (inserts, updates, deletes) executed during a test would be permanently committed to the database. This leads to "test pollution," where the persistent state left behind by one test alters the outcome of subsequent tests. This results in flaky, unpredictable, and order-dependent test failures. When annotated with `@Transactional`, the Spring TestContext framework automatically rolls back the transaction at the end of each test method, ensuring a clean and predictable database state for every single test.

### 5. What does RFC 9457 define, and why does following it produce better APIs than returning a generic `{ error: 'something went wrong' }` message?
**RFC 9457** (which obsoletes RFC 7807) defines "Problem Details for HTTP APIs," a standardized JSON format (`application/problem+json`) for reporting errors. Following this RFC produces better APIs because it provides a predictable, standard schema (including fields like `type`, `title`, `status`, `detail`, and `instance`) that API clients can universally understand and program against. A generic error message forces clients to write brittle, custom parsing logic for every different API they consume. RFC 9457 allows clients to handle errors robustly, display standardized user-friendly messages, or execute specific retry logic based on the structured problem type.

### 6. What is the difference between an integration test (MockMvc) and a unit test (Mockito)? When is each approach preferable?
- **Unit Test (Mockito)**: Focuses on testing a single class or method in complete isolation. Dependencies (like repositories or other services) are mocked using tools like Mockito. They are extremely fast to run and are preferable for thoroughly testing complex business logic, algorithms, and edge cases without the overhead of loading the Spring context or connecting to a database.
- **Integration Test (MockMvc)**: Tests how multiple layers of the application (e.g., Controller, Service, and Repository) interact together. Using `MockMvc`, we simulate HTTP requests and verify routing, JSON serialization/deserialization, validation, and database interactions, typically involving the Spring application context and an in-memory database. They are preferable for validating the end-to-end flow of an API endpoint to ensure all components are wired correctly, security is applied, and the system functions as a cohesive unit.

### 7. What is the purpose of the Service layer? Why should a Controller never call a Repository directly?
The Service layer encapsulates the core business logic of the application. It acts as an intermediary between the Controller (presentation layer) and the Repository (data access layer). A Controller should never call a Repository directly because it tightly couples the web request handling logic with the data access logic, making the code harder to test, reuse, and maintain. By funneling requests through the Service layer, you ensure that business rules, complex validations, and transaction boundaries (`@Transactional`) are consistently applied before data is persisted or retrieved.

### 8. Why do enterprise systems use layered architecture? What problems does it solve compared to writing all logic in one class?
Enterprise systems use layered architecture (typically Presentation, Business Logic, and Data Access) to enforce the separation of concerns. If all logic were written in a single "god class," the code would become a tangled, unmaintainable mess that is impossible to unit test, hard to collaborate on, and difficult to change. Layered architecture solves these problems by improving **maintainability** (you can change the database layer without affecting the web layer), **testability** (layers can be mocked to test in isolation), and **reusability** (the same business service can be consumed by both a REST API and a message queue consumer).

### 9. What advantages does Spring Data JPA provide over writing plain JDBC code? Give at least two specific examples.
Spring Data JPA dramatically reduces boilerplate code and simplifies data access compared to plain JDBC.
- **Automatic CRUD Operations:** Instead of manually managing connections, `PreparedStatement`s, and `ResultSet` mapping, you simply extend an interface like `JpaRepository` and instantly inherit working methods like `save()`, `findAll()`, and `deleteById()`.
- **Query Derivation:** Spring Data JPA can automatically generate SQL queries directly from method names in your repository interface. For example, declaring a method named `findByNameAndPriceLessThan(String name, BigDecimal price)` will automatically generate and execute the correct SQL query without you writing any actual SQL.

### 10. In your unit test, you used a mock for `ProductRepository`. Why is it important to isolate the class under test from its real dependencies?
Isolating the class under test ensures that you are only validating the logic within that specific component. If a unit test for a Service used a real `ProductRepository` connecting to a real database, a test failure might be caused by a database connection error, a constraint violation, or stale data, rather than a bug in the Service logic. Mocking the repository gives you complete control over its behavior (e.g., forcing it to return specific data or throw an exception), making tests deterministic, incredibly fast, and tightly focused.

### 11. What would happen if `spring.jpa.hibernate.ddl-auto` were set to `update` instead of `create-drop`? When would you use each setting?
If set to `update`, Hibernate will inspect the existing database schema on startup and attempt to alter it (e.g., adding new tables or columns) to match your current JPA entity definitions. Existing data is preserved. If set to `create-drop`, Hibernate creates the schema on startup and completely drops all tables when the application shuts down, destroying all data.
- **`create-drop`** is ideal for automated integration tests or scratchpad local development where you want a guaranteed clean slate every time the application runs.
- **`update`** is convenient during early, rapid development when you are frequently modifying entities and want to keep your test data between restarts. (Note: In production environments, this should typically be set to `validate` or `none`, relying on tools like Flyway or Liquibase for safe schema migrations).
